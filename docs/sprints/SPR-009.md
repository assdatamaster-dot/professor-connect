# SPR-009 — Heartbeat & Connection Recovery

## Objetivo

Tornar a camada Socket.IO capaz de detectar clientes inativos, expirar conexões e recuperar o
estado técnico de um cliente que retorna dentro de uma janela configurável. Toda a infraestrutura
é mantida em memória e não inclui WebRTC, mídia, autenticação, persistência ou regras de chamada.

## Arquitetura

```text
packages/protocol/src/heartbeat.ts
                │
services/backend/config/src/environment.ts
                │
services/backend/services/src/modules/heartbeat/
├── heartbeat.events.ts
├── heartbeat.manager.ts
├── heartbeat.service.ts
└── heartbeat.types.ts
                │
services/backend/websocket/src/modules/communication/
├── communication.gateway.ts
├── communication.service.ts
└── communication.types.ts
```

- `HeartbeatManager` mantém registros e índices em `Map`, calcula inatividade, janela de
  reconexão e timeout sem depender de transporte.
- `HeartbeatService` agenda o ciclo periódico e coordena portas pequenas para Connection,
  Presence, Session, Request e Call.
- `CommunicationGateway` recebe `heartbeat.pong`, trata desconexões e traduz eventos internos
  para o protocolo Socket.IO.
- `CommunicationService` cria e envia os `SocketMessage<T>` de heartbeat.
- `services/backend/config` lê e valida todas as variáveis de ambiente.

## Modelo de conexão

`clientId` identifica o cliente de forma estável. `connectionId` é o ID transitório gerado pelo
Socket.IO. Cada registro contém ainda `connectedAt`, `lastSeen`, `ConnectionStatus` e
`ConnectionState`.

- `CONNECTED`: socket registrado e operante;
- `LOST`: transporte desconectado, mas o estado ainda pode ser recuperado;
- `RECOVERED`: novo socket assumiu o registro existente;
- `TIMED_OUT`: limites excedidos e recursos técnicos liberados.

O status técnico usa `ACTIVE`, `INACTIVE` e `CLOSED`. Ele não substitui os status funcionais de
Presence (`ONLINE`, `AVAILABLE`, `BUSY`, `OFFLINE`).

## Fluxo de heartbeat

```text
Scheduler
   ↓ a cada HEARTBEAT_INTERVAL_MS
heartbeat.ping ───────────────→ cliente
heartbeat.pong ←───────────────┘
   ↓
lastSeen atualizado
```

Sem confirmação, o registro torna-se `INACTIVE`. Ao alcançar `HEARTBEAT_TIMEOUT_MS`, o service:

1. marca a conexão como `TIMED_OUT` e a remove;
2. altera Presence para `OFFLINE`;
3. remove a associação do socket das Sessions;
4. encerra o socket, se ele ainda existir;
5. emite `connection.timeout` e registra o timeout.

## Fluxo de reconexão

```text
disconnect
   ↓
connection.lost + estado preservado
   ↓
novo socket envia presence.register com o mesmo clientId
   ├── dentro da janela → connection.recovered + snapshot restaurado
   └── fora da janela   → timeout; registro posterior é uma conexão nova
```

Uma recuperação válida:

- mantém um único registro para o `clientId`;
- substitui o `connectionId` antigo pelo novo;
- preserva display name, role e status de Presence;
- substitui o ID técnico nas Sessions e reinscreve o socket nas salas;
- lista Requests pendentes e Calls não terminais relacionadas ao cliente;
- entrega todo o resultado no payload de `connection.recovered`.

Requests e Calls usam `clientId`, portanto não precisam ser copiadas nem ter seus estados
alterados. No timeout, esses domínios continuam seguindo suas próprias máquinas de estado.

## Eventos

| Evento                 | Direção             | Payload                      |
| ---------------------- | ------------------- | ---------------------------- |
| `heartbeat.ping`       | Servidor → cliente  | `HeartbeatPingPayload`       |
| `heartbeat.pong`       | Cliente → servidor  | `HeartbeatPongPayload`       |
| `connection.lost`      | Servidor → clientes | `ConnectionLifecyclePayload` |
| `connection.recovered` | Servidor → cliente  | `ConnectionRecoveryPayload`  |
| `connection.timeout`   | Servidor → clientes | `ConnectionLifecyclePayload` |

Todos pertencem a `EventType` e usam `SocketMessage<T>`. Nenhuma comunicação paralela ou payload
sem tipagem foi criada.

## Configurações

| Variável                | Padrão  | Regra                                      |
| ----------------------- | ------- | ------------------------------------------ |
| `HEARTBEAT_INTERVAL_MS` | `30000` | inteiro positivo e menor que o timeout     |
| `HEARTBEAT_TIMEOUT_MS`  | `90000` | inteiro positivo                           |
| `RECONNECT_WINDOW_MS`   | `90000` | inteiro positivo e não maior que o timeout |

A janela efetiva termina no primeiro limite alcançado entre o timeout desde `lastSeen` e a janela
desde a desconexão.

## Logs

São registrados: `Heartbeat enviado`, `Heartbeat recebido`, `Cliente inativo`, `Timeout`,
`Reconexão` e `Recuperação concluída`. Os logs incluem somente identificadores e contexto técnico.

## Testes

Os testes de serviço usam relógio e portas controladas para cobrir heartbeat normal, perda,
timeout, recuperação antes do limite, rejeição após o limite, restauração do snapshot e ausência
de duplicidade. O teste Socket.IO cobre desconexão real, `connection.lost`, troca de socket,
restauração de Session e `connection.timeout`.

Execute:

```bash
npm run test --workspace=@professor-connect/services
npm run test --workspace=@professor-connect/websocket
npm run check
npm run build
```
