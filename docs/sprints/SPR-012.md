# SPR-012 — Audio & Video Engine

## Objetivo

Adicionar captura, transmissão, recebimento e renderização de áudio e vídeo entre exatamente dois
clientes, reutilizando signaling, configuração ICE e negociação WebRTC das sprints anteriores.
DataChannel permanece funcional e independente.

Esta sprint não implementa compartilhamento de tela, controle remoto, gravação, transferência de
arquivos, chat ou múltiplos participantes.

## Arquitetura RTC Engine

O repositório possui dois clientes desktop que compartilham `@professor-connect/engine`. Para não
duplicar implementação nem criar um terceiro workspace, o módulo lógico `client/src/core/rtc` fica
em `packages/engine/src/client/core/rtc`:

```text
apps/student-desktop ─┐
                     ├──> @professor-connect/engine
apps/teacher-desktop ─┘            │
                                   └── src/client/core/rtc/
                                       ├── rtc-engine.ts
                                       ├── peer-manager.ts
                                       ├── media-manager.ts
                                       ├── rtc-events.ts
                                       └── rtc-types.ts
                                              │
                     modules/webrtc + State Machine + WebRtcSignalingPort
                                              │
                              signaling server da Sprint 10
```

### RTC Engine

`RtcEngine` é a fachada usada pela interface. Ele expõe configuração de mídia, enumeração de
dispositivos, conexão, recepção de Offer/Answer/ICE, reconexão e encerramento. Também coordena os
renderizadores local/remoto e eventos de lifecycle. Nenhuma API da interface recebe
`RTCPeerConnection`.

### Peer Manager

`PeerManager` compõe a infraestrutura existente:

- `PeerConnectionFactory` cria o peer nativo por uma porta;
- `WebRtcService` executa Offer, Answer e Trickle ICE;
- `WebRtcManager` mantém a Negotiation State Machine;
- `WebRtcSignalingPort` envia os eventos da Sprint 10;
- na reconexão, um runtime completo é encerrado e recriado para a mesma Call/Session.

### Media Manager

`MediaManager` implementa `MediaServicePort` e concentra:

- solicitação de câmera e microfone;
- validação de uma track de áudio e uma de vídeo;
- criação e retenção do MediaStream local;
- adição e remoção de tracks no peer;
- encerramento das tracks;
- enumeração de dispositivos;
- renderização por `RtcMediaRendererPort`.

`BrowserVideoRenderer` é o adaptador para `HTMLVideoElement`. A interface fornece os elementos
local e remoto ao compor o engine; depois disso, interage somente com `RtcEngine`.

## Fluxo do MediaStream

```text
Interface
   │ RtcEngine.connect(callId, sessionId)
   ▼
MediaManager ── getUserMedia(audio + video) ──> MediaStream local
   │                                                  │
   │ render local                                     │ addTrack
   ▼                                                  ▼
HTMLVideoElement local                         RTCPeerConnection
                                                      │
                              Offer / Answer / ICE via signaling
                                                      │
                                               peer remoto
                                                      │ ontrack
                                                      ▼
                                              MediaStream remoto
                                                      │
                                                      ▼
                                          HTMLVideoElement remoto
```

O backend nunca recebe os MediaStreams. Somente SDP e candidatos ICE passam pelo signaling; áudio
e vídeo trafegam peer-to-peer.

## Permissões

`MediaManager` chama `getUserMedia` antes de criar a Offer ou Answer:

1. em sucesso, valida áudio/vídeo e registra `Permissão concedida`, `Câmera iniciada`, `Microfone
iniciado` e `Stream local criada`;
2. `NotAllowedError`/`PermissionDeniedError` registra `Permissão negada` e interrompe a conexão;
3. falta de dispositivo, track ou outra falha registra `Falhas` e libera qualquer track já criada.

Nenhuma credencial, SDP, candidato ICE ou identificador físico do dispositivo é registrado.

## Configuração de dispositivos

`RtcMediaSettings` prepara seleção futura sem acoplar a interface a `MediaStreamConstraints`:

```ts
interface RtcMediaSettings {
  readonly audio: { readonly deviceId?: string };
  readonly video: {
    readonly deviceId?: string;
    readonly width?: number;
    readonly height?: number;
    readonly frameRate?: number;
  };
}
```

Os campos são opcionais. Sem valores explícitos, os dispositivos padrão são usados. Resolução e
FPS devem ser positivos. `listDevices()` retorna somente dispositivos tipados de entrada/saída de
áudio e entrada de vídeo.

## Reconexão e encerramento

`RtcEngine.reconnect()` limpa os renderizadores, encerra o runtime atual e solicita ao
`PeerManager` um novo peer. Uma nova captura e negociação completa são realizadas. O respondente
detecta a nova Offer, encerra seu runtime anterior e também cria um peer limpo.

No encerramento, senders são removidos, tracks são paradas, peers são fechados, listeners do
runtime são descartados e elementos de vídeo recebem `srcObject = null`.

## Eventos e logs

Eventos locais ficam centralizados em `RtcEventType`: stream local criada, stream remota recebida,
peer conectado, reconectando, reconectado, encerrado e falha.

Logs cobertos: `Permissão concedida`, `Permissão negada`, `Câmera iniciada`, `Microfone iniciado`,
`Peer conectado`, `Stream local criada`, `Stream remota recebida`, `Encerramento` e `Falhas`.

## Como executar os testes

O teste usa dois `RTCPeerConnection` reais via `@roamhq/wrtc`, com sources sintéticas de áudio e
vídeo. Ele valida permissões, constraints, captura em ambos os clientes, Offer/Answer/ICE, tracks
remotas, renderização local/remota, reconexão com nova captura e encerramento de todas as tracks.

```bash
npm run test --workspace=@professor-connect/engine
npm run check
npm run build
```

## Limites

Não há compartilhamento de tela, controle remoto, gravação, transferência de arquivos, chat ou
múltiplos participantes. O engine mantém uma única conexão ativa por instância.
