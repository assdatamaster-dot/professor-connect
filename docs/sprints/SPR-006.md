# SPR-006 — Service Request System

## Objetivo

Implementar exclusivamente o ciclo de vida de solicitações de atendimento entre alunos e
professores disponíveis. A sprint não cria sessões, chamadas, áudio, vídeo, WebRTC,
compartilhamento de tela, controle remoto, autenticação ou persistência.

## Arquitetura

O domínio foi adicionado ao workspace de serviços e permanece independente de Socket.IO:

```text
packages/protocol/src/
└── request.ts

services/backend/services/src/modules/request/
├── request.events.ts
├── request.manager.ts
├── request.service.ts
├── request.store.ts
└── request.types.ts

services/backend/websocket/tests/
├── request.spec.ts
└── request-client.ts
```

- `RequestStore` mantém Requests, destinatários e rejeições em memória.
- `RequestManager` gera UUIDs e protege as transições do ciclo de vida.
- `RequestService` valida os papéis por meio de `PresenceService`, controla os timeouts e publica
  expirações por uma interface de assinatura independente do transporte.
- `CommunicationGateway` valida os envelopes e traduz eventos Socket.IO em chamadas ao service.
- `CommunicationService` cria `SocketMessage<T>` e entrega notificações somente às conexões
  envolvidas.

Essa separação mantém o domínio sem dependência de Socket.IO, configuração de ambiente ou banco
de dados. O timeout é lido exclusivamente pelo workspace `services/backend/config` e injetado na composição
do servidor.

## Modelo

Cada `ServiceRequest` possui:

- `requestId`: UUID gerado no servidor;
- `studentId`: cliente técnico que criou a solicitação;
- `teacherId`: ausente enquanto pendente e definido no aceite;
- `status`: `PENDING`, `ACCEPTED`, `REJECTED`, `CANCELLED` ou `EXPIRED`;
- `createdAt`: criação em UTC/ISO 8601;
- `acceptedAt`: definido somente no aceite;
- `expiresAt`: instante previsto para expiração em UTC/ISO 8601.

O protocolo inclui `REJECTED` entre os estados possíveis, mas a rejeição desta sprint pertence a
cada destinatário. Por isso, rejeitar registra o professor e emite `request.rejected`, enquanto a
Request compartilhada permanece `PENDING`. Isso preserva a possibilidade de outro professor
aceitar e cumpre a regra de manter o pedido pendente mesmo quando todos rejeitam.

## Eventos

Todos os eventos pertencem a `EventType` e todas as mensagens usam `SocketMessage<T>`:

| Evento              | Comportamento                                            |
| ------------------- | -------------------------------------------------------- |
| `request.create`    | Aluno conectado solicita atendimento                     |
| `request.created`   | Servidor confirma a criação ao aluno                     |
| `request.received`  | Professores `AVAILABLE` no momento da criação recebem    |
| `request.accept`    | Um professor destinatário aceita                         |
| `request.accepted`  | Aluno e todos os destinatários recebem o resultado       |
| `request.reject`    | Um professor destinatário rejeita individualmente        |
| `request.rejected`  | Professor recebe confirmação da própria rejeição         |
| `request.cancel`    | Aluno criador cancela uma Request ainda pendente         |
| `request.cancelled` | Aluno e professores destinatários recebem o cancelamento |
| `request.expired`   | Aluno e professores conectados recebem a expiração       |

## Fluxos implementados

### Criação

1. O gateway identifica o aluno pela presença vinculada à conexão.
2. O service obtém um snapshot dos professores `AVAILABLE`.
3. O manager gera o UUID, calcula `expiresAt` e armazena a Request como `PENDING`.
4. O aluno recebe `request.created` e os professores recebem `request.received`.
5. Um timeout cancelável é agendado.

### Aceite

1. Somente um professor que recebeu a Request pode aceitar.
2. O manager define `teacherId`, `acceptedAt` e status `ACCEPTED`.
3. O timeout é cancelado.
4. Aluno, professor que aceitou e demais destinatários recebem `request.accepted`; os demais
   professores podem remover a oferta de suas interfaces.

### Rejeição

Cada professor pode rejeitar uma única vez. A rejeição é armazenada separadamente e confirmada
por `request.rejected`. A Request permanece ativa e `PENDING`, mesmo depois da rejeição de todos
os destinatários, até que seja aceita, cancelada ou expire.

### Cancelamento e expiração

Somente o aluno criador pode cancelar antes do aceite. O cancelamento define `CANCELLED`, encerra
o timer e notifica os envolvidos. O timeout padrão é de 60 segundos e pode ser configurado por
`REQUEST_TIMEOUT_MS`; ao disparar, altera Requests ainda pendentes para `EXPIRED`, remove-as da
lista ativa e notifica os clientes que continuarem conectados.

## Como testar

Execute todas as validações:

```bash
npm install
npm run check
npm run build
```

O teste de integração conecta três professores e dois alunos por Socket.IO e cobre criação,
aceite, três rejeições, cancelamento, expiração, notificações, armazenamento e logs.

Para a simulação manual, configure um timeout curto no `.env`:

```dotenv
REQUEST_TIMEOUT_MS=1000
```

Em dois terminais, execute:

```bash
npm run dev
npm run request:client
```

Sem alteração da variável, a expiração usa o padrão de 60 segundos.
