# SPR-008 — Call Manager

## Objetivo

Implementar o gerenciamento em memória do ciclo de vida de Calls. Uma Call nasce após o aceite de
uma Request, identifica aluno e professor e pode avançar por estados técnicos de preparação,
conexão simulada e encerramento.

Esta sprint não implementa WebRTC, áudio, vídeo, compartilhamento de tela, controle remoto,
autenticação, interface gráfica ou persistência.

## Arquitetura

O módulo foi adicionado ao workspace de serviços existente:

```text
packages/protocol/src/
└── call.ts

services/backend/services/src/modules/call/
├── call.events.ts
├── call.manager.ts
├── call.service.ts
├── call.state-machine.ts
├── call.store.ts
└── call.types.ts
```

- `CallStore` implementa criação, busca, atualização, remoção e listagem com `Map`.
- `CallStateMachine` configura o grafo específico usando `StateMachine<TState>` da Sprint 7.
- `CallManager` gera UUIDs, mantém uma máquina por Call, executa transições e expõe histórico.
- `CallService` valida a Request aceita, coordena o manager, registra logs e publica eventos de
  ciclo de vida por listeners independentes do transporte.
- `CommunicationGateway` cria a Call depois do aceite e traduz eventos internos para Socket.IO.
- `CommunicationService` cria `SocketMessage<CallPayload>` e notifica somente aluno e professor.

O `CallService` depende de uma interface pequena `AcceptedRequestReader`, não da implementação
interna de Request. Isso permite testes isolados e mantém baixo acoplamento.

## Modelo

Cada Call possui:

- `callId`: UUID gerado no servidor;
- `requestId`: Request aceita de origem;
- `sessionId`: associação opcional, que também pode ser definida depois da criação;
- `studentId` e `teacherId`: copiados da Request aceita;
- `status`: estado controlado pela máquina;
- `createdAt`: criação em UTC/ISO 8601;
- `connectedAt`: definido ao entrar em `CONNECTED`;
- `finishedAt`: definido ao entrar em `FINISHED`, `FAILED` ou `CANCELLED`.

Uma Request pode originar somente uma Call no store em memória.

## Máquina de estados

```text
CREATED ─────> CONNECTING ─────> CONNECTED ─────> FINISHED
   │               │
   ├──> FAILED     ├──> FAILED
   └──> CANCELLED  └──> CANCELLED
```

Transições permitidas:

| Estado anterior | Novo estado  |
| --------------- | ------------ |
| `CREATED`       | `CONNECTING` |
| `CREATED`       | `FAILED`     |
| `CREATED`       | `CANCELLED`  |
| `CONNECTING`    | `CONNECTED`  |
| `CONNECTING`    | `FAILED`     |
| `CONNECTING`    | `CANCELLED`  |
| `CONNECTED`     | `FINISHED`   |

Todas as outras 29 combinações são inválidas e lançam `InvalidStateTransitionError`. Nenhum
estado é atribuído diretamente: `CallManager` usa sempre `transition.nextState`, e os timestamps
de conexão e finalização usam o mesmo `transition.timestamp` registrado no histórico.

## Fluxo integrado

1. Um professor destinatário aceita uma Request pendente.
2. `RequestStateMachine` valida `PENDING → ACCEPTED`.
3. O gateway chama `CallService.createCall(requestId)`.
4. O service confirma que a Request está `ACCEPTED` e possui `teacherId`.
5. O manager cria a Call em `CREATED` com UUID.
6. Aluno e professor aceito recebem `call.created`.
7. O gateway solicita `startCall`, validando `CREATED → CONNECTING`.
8. Aluno e professor recebem `call.connecting`.
9. Futuros adaptadores poderão chamar `connectCall`, `finishCall`, `failCall` ou `cancelCall`.

Os demais professores recebem `request.accepted` para remover a oferta, mas não recebem eventos da
Call.

## Eventos

Todos os nomes pertencem a `EventType` e as notificações usam `SocketMessage<CallPayload>`:

- `call.create`;
- `call.created`;
- `call.connecting`;
- `call.connected`;
- `call.finished`;
- `call.cancelled`;
- `call.failed`.

`CallService` publica `call.created` na criação e um evento correspondente para cada transição
válida. O gateway assina esses eventos sem introduzir dependência de Socket.IO no domínio.

## Logs

- `Call criada`;
- `Call iniciada`;
- `Mudança de estado`, produzido pela State Machine;
- `Falha`;
- `Cancelamento`;
- `Finalização`.

## Testes

Os testes de serviço cobrem:

- criação a partir de Request aceita;
- associação posterior de `sessionId`;
- `CREATED → CONNECTING → CONNECTED → FINISHED`;
- falha a partir de `CREATED` e `CONNECTING`;
- cancelamento a partir de `CREATED` e `CONNECTING`;
- histórico e timestamps;
- store, remoção, eventos e logs;
- todas as 29 transições inválidas.

O teste Socket.IO de Requests confirma também a criação automática e as notificações
`call.created` e `call.connecting`.

Execute:

```bash
npm run test --workspace=@professor-connect/services
npm run check
npm run build
```

## Integração futura com WebRTC

Um adaptador futuro poderá preparar negociação WebRTC enquanto a Call estiver `CONNECTING` e
chamar `connectCall` após a confirmação técnica. Encerramentos normais usarão `finishCall`; falhas
de negociação usarão `failCall`. Nenhum SDP, ICE candidate, stream ou dispositivo de mídia faz
parte desta sprint.
