# SPR-007 — Request State Machine

## Objetivo

Adicionar uma máquina de estados genérica e utilizá-la como única responsável pelas mudanças de
status de Requests. A infraestrutura impede transições inválidas, registra histórico, emite
eventos em memória e pode receber novos grafos no futuro sem depender de Sessions, Calls,
compartilhamento de tela ou controle remoto.

A sprint não implementa essas capacidades futuras, nem WebRTC, áudio, vídeo, autenticação, JWT ou
persistência.

## Localização arquitetural

O repositório é um monorepo e não possui um workspace em `backend/src`. Criar código diretamente
nesse caminho o deixaria fora dos scripts de build, lint e typecheck. Seguindo a arquitetura
existente e o `DEVELOPER_GUIDE`, a infraestrutura foi colocada no workspace proprietário das
regras de aplicação:

```text
backend/services/src/
├── core/state-machine/
│   ├── state-machine.ts
│   ├── state-transition.ts
│   ├── state-machine.types.ts
│   └── state-machine.errors.ts
└── modules/request/
    └── request-state-machine.ts
```

Não houve criação de workspace, reorganização de módulos ou alteração de tecnologia.

## Infraestrutura genérica

`StateMachine<TState>` depende somente de tipos genéricos e recebe:

- estado inicial;
- lista de pares `from`/`to` permitidos;
- relógio opcional;
- logger opcional;
- contexto técnico opcional para logs.

Ela expõe:

- `getCurrentState()` para consultar o estado atual;
- `canTransitionTo()` para consultar o grafo sem mudar estado;
- `transitionTo()` para validar e executar uma transição;
- `getHistory()` para obter uma cópia readonly do histórico;
- `onTransition()` para assinar eventos de mudanças válidas.

Uma mudança válida cria `StateTransition<TState>` contendo `previousState`, `nextState` e
`timestamp`. Em seguida, a máquina atualiza o estado, registra o histórico, registra o log
`Mudança de estado` e notifica os listeners.

Uma mudança inválida registra `Tentativa inválida` e `Erro de transição`, preserva estado e
histórico, não emite evento e lança `InvalidStateTransitionError`. O erro possui o código estável
`INVALID_STATE_TRANSITION` e informa os estados de origem e destino.

## RequestStateMachine

O adaptador específico configura o seguinte grafo:

```text
             ┌──> ACCEPTED
             ├──> REJECTED
PENDING ─────┼──> CANCELLED
             └──> EXPIRED
```

As únicas transições válidas são:

| Estado anterior | Novo estado |
| --------------- | ----------- |
| `PENDING`       | `ACCEPTED`  |
| `PENDING`       | `REJECTED`  |
| `PENDING`       | `CANCELLED` |
| `PENDING`       | `EXPIRED`   |

Todos os estados de destino são terminais nesta sprint. Autotransições, retornos a `PENDING` e
qualquer transição a partir de `ACCEPTED`, `REJECTED`, `CANCELLED` ou `EXPIRED` são inválidos.

## Integração com Requests

Cada Request criada pelo `RequestManager` recebe sua própria `RequestStateMachine` em memória.
O manager não atribui diretamente estados terminais:

1. solicita a mudança à máquina específica da Request;
2. recebe um objeto `StateTransition` após a validação;
3. atualiza o store usando `transition.nextState`;
4. no aceite, usa `transition.timestamp` também como `acceptedAt`;
5. disponibiliza o histórico por `getStateHistory()`.

O timeout verifica se a Request continua pendente antes de solicitar expiração, evitando uma
tentativa concorrente depois de aceite ou cancelamento. O timer continua cancelável e armazenado
somente em memória.

As rejeições individuais de professores continuam sendo metadados de destinatário conforme a
Sprint 6 e não alteram o estado compartilhado. A transição compartilhada para `REJECTED` existe no
grafo e é validada pela máquina, mas esta sprint não adiciona uma nova regra de produto para
acioná-la.

## Eventos e logs

A máquina emite um evento interno para cada mudança válida. O gateway continua emitindo os
eventos Socket.IO correspondentes a aceite, cancelamento e expiração depois que o service retorna
uma Request já validada.

Logs da máquina:

- `Mudança de estado` para transição válida;
- `Tentativa inválida` antes de rejeitar uma transição;
- `Erro de transição` com o erro controlado.

Falhas de listeners são isoladas e registradas sem desfazer uma transição já concluída.

## Testes

Os testes de `backend/services` cobrem:

- as quatro transições válidas;
- todas as 21 combinações inválidas do grafo;
- preservação do estado após erro;
- histórico e timestamp;
- emissão de evento somente para mudança válida;
- logs de sucesso e erro;
- uso da máquina genérica com um segundo conjunto de estados.

O teste Socket.IO da Sprint 6 também valida que aceite, cancelamento e expiração produzem os
respectivos históricos.

Para executar apenas os testes da máquina:

```bash
npm run test --workspace=@professor-connect/services
```

Para validar todo o monorepo:

```bash
npm run check
npm run build
```
