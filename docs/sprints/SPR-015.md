# SPR-015 — MVP Integration & Stabilization

## Objetivo

Integrar o ciclo completo do atendimento remoto sem mover ou reimplementar os módulos das Sprints
anteriores. A Sprint adiciona somente orquestração, verificação de saúde, recuperação e liberação
coordenada de recursos.

Continuam fora do escopo execução real de mouse/teclado, arquivos, gravação, chat, múltiplos
participantes e painel administrativo.

## Arquitetura integrada

O módulo lógico `client/src/core/workflow` segue o padrão dos demais núcleos compartilhados:

```text
packages/engine/src/client/core/workflow/
├── workflow.manager.ts       # Orquestra o ciclo e o contexto
├── workflow.service.ts       # Fachada do MVP
├── workflow.events.ts        # Estados, transições e eventos locais
├── workflow.types.ts         # Portas e snapshots tipados
├── resource.manager.ts       # Teardown completo e tolerante a falhas
└── health-check.service.ts   # Saúde integrada da conexão
```

Os aplicativos de aluno e professor exportam as mesmas fachadas pelo workspace
`@professor-connect/engine`. A apresentação não acessa PeerConnection ou serviços do backend
diretamente.

```text
Apresentação
    │
WorkflowService
    │
WorkflowManager ───── HealthCheckService
    │                         │
    ├── Connection/Presence/Heartbeat
    ├── Request/Session/Call
    ├── Signaling/WebRTC/DataChannel
    ├── Screen Sharing/Remote Control
    └── ResourceManager ── teardown por interfaces
```

As portas do Workflow são fronteiras de composição. Implementações existentes continuam donas de
estado, protocolo e regras; adapters apenas convertem suas APIs para a operação exigida pelo fluxo.

## Fluxo completo

```text
1.  Cliente conecta
2.  Presence é registrada/atualizada
3.  Aluno cria Request
4.  Professor aceita Request
5.  Session é criada e recebe os dois participantes
6.  Call é criada e associada à Session
7.  Listeners de Signaling são preparados
8.  WebRTC conecta
9.  DataChannel conecta
10. Áudio e vídeo ficam disponíveis
11. Screen Sharing pode ser iniciado
12. Remote Control pode ser autorizado e transportar comando
13. Call é finalizada
14. Session é encerrada
15. Recursos técnicos, timers, listeners e memória são liberados
```

`begin()` cobre as etapas 1–3. `accept()` cobre 4–10 e só entra em `ACTIVE` depois de um Health
Check saudável. Screen Sharing e Remote Control permanecem ações explícitas. `end()` executa o
teardown e retorna `ResourceReleaseReport`.

## Workflow Manager

Estados:

- `IDLE`;
- `CONNECTING`;
- `REQUESTED`;
- `PREPARING`;
- `NEGOTIATING`;
- `ACTIVE`;
- `RECOVERING`;
- `STOPPING`;
- `COMPLETED`;
- `FAILED`.

```text
IDLE → CONNECTING → REQUESTED → PREPARING → NEGOTIATING → ACTIVE
                                                        ↕
                                                   RECOVERING
                                                        │
ACTIVE → STOPPING → COMPLETED                            └→ FAILED

COMPLETED | FAILED → CONNECTING
```

As transições usam a infraestrutura genérica `StateMachine`. Cada atendimento recebe um novo
`workflowId`, mantém timestamps de início da sessão e da Call e registra suas durações no final.

## Recuperação

Uma queda durante `ACTIVE` move o workflow para `RECOVERING`. O manager solicita recuperação dos
sockets, reconexão do RTC Engine e reconexão do DataChannel. O estado volta a `ACTIVE` somente se
os sete componentes do Health Check estiverem saudáveis. Qualquer erro leva a `FAILED` e dispara
liberação do contexto parcial.

## Health Check

O snapshot contém resultados individuais para:

- Socket.IO;
- Heartbeat;
- Call;
- Session;
- PeerConnection;
- DataChannel;
- MediaStreams.

O resultado geral é `HEALTHY` apenas quando todos são verdadeiros. Um componente indisponível
produz `UNHEALTHY`, usado tanto no início quanto na recuperação.

## Resource Manager

O teardown tenta, em ordem segura:

1. parar Screen Sharing;
2. revogar Remote Control e cancelar sua autorização;
3. finalizar a Call ativa;
4. encerrar a Session ativa;
5. fechar PeerConnection e MediaStreams pelo adapter do RTC Engine;
6. fechar DataChannel;
7. cancelar timers de Heartbeat e Request;
8. remover listeners de Signaling e listeners registrados;
9. limpar memória do Workflow.

Cada etapa é registrada. Falhas entram em `ResourceReleaseReport`, mas não interrompem as etapas
restantes. Um workflow sem falhas de liberação torna-se `COMPLETED`; com falhas, torna-se `FAILED`
e pode ter o teardown tentado novamente.

## Eventos e logs

Os eventos locais cobrem início, Request, aceite, Session, Call, Signaling, WebRTC, DataChannel,
mídia, Screen Sharing, Remote Control, perda, recuperação, falha, encerramento e recursos
liberados. Eles não criam um novo protocolo de servidor.

São registrados início e fim do atendimento, duração da sessão, duração da chamada, falhas,
recuperações e cada recurso liberado.

## Como executar o MVP

Instale as dependências e prepare o ambiente:

```bash
npm install
Copy-Item .env.example .env
```

Execute o backend:

```bash
npm run dev
```

Em outro terminal, valide o MVP integrado:

```bash
npm run test --workspace=@professor-connect/engine
```

Validação completa do monorepo:

```bash
npm run check
npm run build
```

Os aplicativos desktop ainda não possuem interface visual final. Nesta Sprint, “executar o MVP”
significa executar o backend e a suíte de integração que percorre o workflow pelas mesmas
fachadas exportadas para os clientes.

## Testes

`workflow.spec.ts` demonstra:

- fluxo completo com todos os recursos;
- falha na negociação WebRTC;
- queda e recuperação durante atendimento;
- continuidade da liberação após falha parcial;
- múltiplos atendimentos sequenciais sem reutilização de contexto.

As suítes anteriores continuam validando módulos individuais e transportes reais de Socket.IO e
WebRTC.

## Limites

O Workflow não executa comandos remotos, não introduz persistência, não cria um terceiro cliente,
não agrega participantes e não substitui os state machines existentes.
