# SPR-013 — Screen Sharing Engine

## Objetivo

Permitir que o professor solicite a tela do aluno durante uma Call WebRTC. Após o aceite, o aluno
captura a tela, substitui dinamicamente sua track de câmera e pode encerrar o compartilhamento. A
câmera é restaurada sem recriar ou renegociar a conexão.

Não são implementados controle remoto, desenho, gravação, transferência de arquivos, chat,
múltiplos participantes ou múltiplos compartilhamentos simultâneos.

## Arquitetura

O módulo lógico `client/src/core/rtc` continua localizado no workspace compartilhado:

```text
packages/engine/src/client/core/rtc/
├── rtc-engine.ts                 # Fachada exclusiva da interface
├── peer-manager.ts               # Controla sender e replaceTrack
├── media-manager.ts              # Mantém câmera, microfone e renderização
├── screen-sharing.manager.ts     # Captura, track, preview e State Machine
├── screen-sharing.service.ts     # Request/Event e lifecycle distribuído
├── screen-sharing.types.ts       # Portas e contratos
└── screen-sharing.events.ts      # EventType e grafo de estados
```

`ScreenSharingManager` depende de portas para captura, troca de vídeo, mídia e renderização. Ele não
acessa Socket.IO nem `RTCPeerConnection`. `ScreenSharingService` depende do manager e de
`ScreenSharingSignalingPort`. `RtcEngine` expõe todas as ações para a interface.

## Fluxo de solicitação

```text
Professor                       Relay existente                       Aluno
   │                                  │                                  │
   ├── SCREEN_SHARE_REQUEST ─────────>│─────────────────────────────────>│
   │                                  │                                  │ aceita
   │<─ SCREEN_SHARE_ACCEPT ───────────┤<─────────────────────────────────┤
   │                                  │                   getDisplayMedia │
   │<─ SCREEN_SHARE_STARTED ──────────┤<─────────────────────────────────┤
   │<════════════ vídeo de tela na conexão WebRTC existente ════════════│
   │                                  │                         onended  │
   │<─ SCREEN_SHARE_STOPPED ──────────┤<─────────────────────────────────┤
   │<════════════ vídeo da câmera restaurado automaticamente ═══════════│
```

O request usa o mesmo Socket.IO, `SignalingGateway`, `SignalingManager`, Session e Call do
signaling existente. `requestId` correlaciona aceite, recusa e lifecycle. O servidor valida os dois
participantes e apenas encaminha envelopes; não captura ou armazena mídia.

## Captura e troca da track

1. O aluno aceita a solicitação.
2. O manager chama `getDisplayMedia({ video: true, audio: false })`.
3. A primeira track de vídeo da tela recebe um handler de `ended`.
4. `PeerManager.replaceVideoTrack()` localiza o `RTCRtpSender` da câmera.
5. `RTCRtpSender.replaceTrack()` troca câmera por tela sem Offer/Answer adicional.
6. O preview local passa a renderizar o MediaStream de tela.
7. Ao encerrar, o mesmo sender recebe novamente a track da câmera e o preview local é restaurado.

O microfone capturado na Sprint 12 permanece no peer. A captura de áudio da tela fica desativada.

## State Machine

Estados:

- `IDLE`;
- `REQUESTED`;
- `STARTING`;
- `SHARING`;
- `STOPPING`;
- `STOPPED`;
- `FAILED`.

```text
IDLE ──> REQUESTED ──> STARTING ──> SHARING ──> STOPPING ──> STOPPED
             │             │            │            │
             └─────────────┴────────────┴────────────┴──────────> FAILED

STOPPED ──> REQUESTED
FAILED  ──> REQUESTED | STOPPED
REQUESTED ──> STOPPED  # recusa
```

Todas as transições passam por `StateMachine<ScreenSharingState>` da infraestrutura genérica. Uma
segunda solicitação enquanto `REQUESTED`, `STARTING`, `SHARING` ou `STOPPING` é rejeitada pela
máquina.

## Eventos e mensagens

| `EventType`            | Payload                       |
| ---------------------- | ----------------------------- |
| `SCREEN_SHARE_REQUEST` | `ScreenShareRequestPayload`   |
| `SCREEN_SHARE_ACCEPT`  | `ScreenShareReferencePayload` |
| `SCREEN_SHARE_DENY`    | `ScreenShareReferencePayload` |
| `SCREEN_SHARE_STARTED` | `ScreenShareReferencePayload` |
| `SCREEN_SHARE_STOPPED` | `ScreenShareReferencePayload` |
| `SCREEN_SHARE_FAILED`  | `ScreenShareFailedPayload`    |

Todos os payloads possuem interfaces nomeadas e viajam em `SocketMessage<T>`. Nenhuma string
literal é usada para selecionar eventos. Falhas distinguem captura e substituição de track sem
transportar detalhes internos ou dados sensíveis.

## Encerramento pelo usuário

Navegadores encerram a track de display quando o usuário seleciona “parar compartilhamento”. O
adapter `BrowserMediaTrack` entrega esse `ended` ao manager. O manager entra em `STOPPING`, restaura
a câmera, encerra todas as tracks do display, entra em `STOPPED` e o service envia
`SCREEN_SHARE_STOPPED` ao professor.

## Logs

São registrados: `Solicitação enviada`, `Solicitação aceita`, `Solicitação recusada`, `Captura
iniciada`, `Captura encerrada`, `Troca de track` e `Falhas`. SDP, candidatos, conteúdo da tela e
identificadores físicos de dispositivos não entram nos logs.

## Como executar os testes

Os testes simulam professor e aluno pela fachada `RtcEngine`, verificam envelopes, estados,
visualização da tela, encerramento pelo usuário, retorno automático à câmera e falha de captura. O
teste Socket.IO comprova que request e accept atravessam o relay existente com validação da Call.

```bash
npm run test --workspace=@professor-connect/engine
npm run test --workspace=@professor-connect/websocket
npm run check
npm run build
```

## Limites

Há apenas um compartilhamento ativo por instância e exatamente dois participantes por Session. Não
há controle remoto, desenho, gravação, transferência de arquivos ou chat.
