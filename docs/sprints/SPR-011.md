# SPR-011 — WebRTC Peer Connection & DataChannel

## Objetivo

Validar a negociação WebRTC completa entre exatamente dois clientes e abrir um DataChannel
peer-to-peer. Esta sprint não solicita câmera ou microfone e não implementa áudio, vídeo,
compartilhamento de tela, controle remoto, gravação, transferência de arquivos ou chat.

## Arquitetura

```text
apps/student-desktop ─┐
                     ├──> packages/engine/
apps/teacher-desktop ─┘    ├── src/config/webrtc.ts
                           └── src/modules/webrtc/
                               ├── peer.factory.ts
                               ├── peer.types.ts
                               ├── peer.events.ts
                               ├── data-channel.service.ts
                               ├── webrtc.manager.ts
                               └── webrtc.service.ts
                                      │
                    @professor-connect/services/state-machine
                                      │
                 WebRtcSignalingPort + SocketMessage<T>
                                      │
                  services/backend/websocket/modules/signaling
```

- `PeerFactory` adapta `RTCPeerConnection` e `RTCDataChannel` nativos a portas pequenas.
- `DataChannelService` abre, fecha, serializa, valida e entrega mensagens do canal.
- `DataChannelWebRtcManager` mantém uma negociação e uma State Machine por `callId`.
- `DataChannelWebRtcService` coordena SDP, Trickle ICE, estado, DataChannel e encerramento.
- `WebRtcSignalingPort` reutiliza Offer, Answer e ICE da Sprint 10 sem duplicar Socket.IO ou as
  validações do servidor.

O canal padrão é `professor-connect-control`. A nova composição é exportada pelos dois aplicativos
desktop. O fluxo não importa nem chama serviços de mídia.

## Fluxo de negociação

```text
Iniciador                      Signaling                     Respondente
   │ Peer + DataChannel           │                               │
   │ createOffer/setLocal         │                               │
   ├──── signal.offer ───────────>├──── signal.offer ────────────>│
   │                              │        setRemote/createAnswer │
   │<─── signal.answer ───────────┤<─── signal.answer ────────────┤
   │ setRemote                    │                               │
   ├── signal.ice-candidate ─────>├── signal.ice-candidate ──────>│
   │<─ signal.ice-candidate ──────┤<─ signal.ice-candidate ───────┤
```

Candidatos locais produzidos antes do envio do SDP ficam em fila. O serviço somente libera a fila
depois de enviar a Offer ou Answer correspondente, preservando a ordem do Trickle ICE.

## Negotiation State Machine

Estados: `NEW`, `CONNECTING`, `NEGOTIATING`, `CONNECTED`, `FAILED` e `CLOSED`.

```text
NEW → CONNECTING → NEGOTIATING → CONNECTED → CLOSED
 └──────── estados ativos ────────────────→ FAILED → CLOSED
 └──────── estados ativos ─────────────────────────→ CLOSED
```

Todas as transições usam `StateMachine<PeerNegotiationState>` da Sprint 7. Cada mudança produz
`SocketMessage<PeerNegotiationStatePayload>` com `EventType.WEBRTC_PEER_STATE_CHANGED`. O estado
`CONNECTED` exige o `connectionState` conectado e o DataChannel aberto.

## Fluxo do DataChannel

```text
Cliente A                                              Cliente B
   │ createDataChannel("professor-connect-control")       │
   │<──────────── evento nativo ondatachannel ────────────>│
   │<════════════════════ canal open ═════════════════════>│
   ├── SocketMessage<DataChannelMessage<...>> ────────────>│
   │<─ SocketMessage<DataChannelMessage<...>> ─────────────┤
   │<════════════════════ close ══════════════════════════>│
```

O lifecycle de abertura é idempotente porque implementações nativas podem expor o estado `open`
no momento do bind e também disparar o callback. Fechamento inesperado ou erro leva a negociação
para `FAILED`; fechamento solicitado leva a `CLOSED`.

## Eventos e payloads

| `EventType`                   | Payload                                  | Transporte    |
| ----------------------------- | ---------------------------------------- | ------------- |
| `SIGNAL_OFFER`                | `SignalOfferPayload { callId, sdp }`     | signaling     |
| `SIGNAL_ANSWER`               | `SignalAnswerPayload { callId, sdp }`    | signaling     |
| `SIGNAL_ICE_CANDIDATE`        | `SignalIceCandidatePayload`              | signaling     |
| `WEBRTC_PEER_STATE_CHANGED`   | `PeerNegotiationStatePayload`            | evento tipado |
| `WEBRTC_DATA_CHANNEL_MESSAGE` | `DataChannelMessage<DataChannelPayload>` | DataChannel   |

Todos usam `SocketMessage<T>` e membros do enum `EventType`; não existem strings literais para
eventos.

O protocolo de dados é tipado:

```ts
interface DataChannelMessage<TPayload> {
  readonly type: DataChannelMessageType;
  readonly timestamp: string;
  readonly payload: TPayload;
}
```

O receptor valida envelope, evento, sessão, timestamps, tipo e payload antes de publicar a
mensagem. Dados inválidos são tratados como erro e não chegam aos listeners.

## Configuração de STUN e TURN

`packages/engine/src/config/webrtc.ts` centraliza `RTCConfiguration` e carrega:

- `WEBRTC_STUN_URLS`: lista de URLs separadas por vírgula;
- `WEBRTC_TURN_ENABLED`: `true` ou `false`;
- `WEBRTC_TURN_URLS`: lista de URLs TURN;
- `WEBRTC_TURN_USERNAME`: usuário opcional;
- `WEBRTC_TURN_CREDENTIAL`: credencial opcional.

O padrão usa `stun:stun.l.google.com:19302` e deixa TURN desabilitado. Se TURN for habilitado, ao
menos uma URL é obrigatória. Os testes desabilitam STUN/TURN para serem locais e determinísticos.

## Logs

São registrados: `Peer criado`, `Offer enviada`, `Offer recebida`, `Answer enviada`, `Answer
recebida`, `ICE Candidate enviado`, `ICE Candidate recebido`, `DataChannel aberto`, `Mensagem
enviada`, `Mensagem recebida`, `DataChannel fechado`, `Peer conectado`, `Peer fechado` e `Erro`.
SDP, candidatos e credenciais não entram no log.

## Como executar os testes

O teste funcional usa dois `RTCPeerConnection` reais fornecidos por `@roamhq/wrtc` no Node 22,
sem tracks de mídia. Ele comprova configuração, Offer, Answer, ICE nos dois sentidos, estados,
abertura do canal, mensagens bidirecionais e encerramento.

```bash
npm run test --workspace=@professor-connect/engine
npm run check
npm run build
```

## Limites da sprint

Não foram implementados áudio, vídeo, captura de mídia, compartilhamento de tela, controle remoto,
gravação, transferência de arquivos, chat ou múltiplos participantes. Não há banco, JWT ou login.
