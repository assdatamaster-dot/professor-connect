# SPR-005 — Client Presence & Availability

## Objetivo

Adicionar presença em memória para identificar clientes online e seus estados de disponibilidade.
A sprint não implementa chamadas, autenticação, persistência, WebRTC, compartilhamento de tela ou
comportamentos específicos de professor e aluno.

## Arquitetura

O módulo foi adicionado a `backend/services/src/modules/presence`, mantendo estado e consultas
independentes de Socket.IO. O `CommunicationGateway` traduz mensagens do protocolo e chama apenas
o `PresenceService`. Os contratos consumidos por servidor e clientes ficam em
`packages/shared-types/src/presence.ts`.

```text
packages/shared-types/src/
└── presence.ts

backend/services/src/modules/presence/
├── presence.manager.ts
├── presence.service.ts
├── presence.types.ts
└── presence.events.ts

backend/websocket/tests/
├── presence.spec.ts
└── presence-client.ts
```

## Modelo de presença

Cada registro contém:

- `clientId`: identificador estável informado pelo cliente;
- `connectionId`: identificador da conexão Socket.IO;
- `displayName`: nome usado para exibição;
- `role`: `STUDENT` ou `TEACHER`;
- `status`: `ONLINE`, `AVAILABLE`, `BUSY` ou `OFFLINE`;
- `lastSeen`: timestamp UTC no formato ISO 8601.

`PresenceManager` mantém dois Maps: presença por `clientId` e associação entre `connectionId` e
`clientId`. `PresenceService` verifica a existência da conexão por meio de `ConnectionService`,
sem criar dependência entre managers.

## Eventos

Todos os eventos são membros de `EventType` e todas as mensagens usam `SocketMessage<T>`:

- `presence.register` registra o cliente conectado;
- `presence.update` altera seu status;
- `presence.online` publica ou consulta clientes online;
- `presence.offline` publica clientes desconectados;
- `presence.available` publica ou consulta professores disponíveis;
- `presence.busy` publica clientes ocupados.

## Fluxo

1. Socket.IO aceita a conexão e `ConnectionService` registra o `connectionId`.
2. O cliente envia `presence.register` com `clientId`, `displayName` e `role`.
3. A presença é criada como `ONLINE`, recebe `lastSeen` e é armazenada em memória.
4. `presence.update` altera o status e atualiza `lastSeen`.
5. Consultas retornam clientes online, professores `AVAILABLE` ou alunos conectados.
6. Ping também atualiza `lastSeen` quando a presença já estiver registrada.
7. Na desconexão, o status passa para `OFFLINE` e a associação da conexão é removida.

## Como testar

Execute as verificações completas:

```bash
npm install
npm run check
npm run build
```

O teste automatizado conecta cinco clientes Socket.IO, registra dois professores e três alunos,
altera os professores para `AVAILABLE` e `BUSY`, consulta a disponibilidade e confirma a remoção
da lista online após a desconexão.

Para executar somente os testes de comunicação:

```bash
npm run test --workspace=@professor-connect/websocket
```

Para executar a simulação manual em dois terminais:

```bash
npm run dev
npm run presence:client
```
