# SPR-004 — Session Manager & Communication Protocol

## Objetivo

Estabelecer a infraestrutura de conexões e sessões em memória e padronizar todas as mensagens de
aplicação transmitidas pelo Socket.IO. A sprint não introduz autenticação, perfis, persistência,
WebRTC ou qualquer regra de negócio de produto.

## Arquitetura

A implementação preserva os workspaces definidos no monorepo:

- `packages/shared-types`: contratos estáveis consumidos pelos serviços, servidor e clientes.
- `backend/services`: estado e operações de conexão e sessão, sem dependência de Socket.IO.
- `backend/websocket`: tradução entre eventos Socket.IO e Services de aplicação.
- `backend/api`: composição do servidor HTTP e injeção do logger.

Os managers não dependem entre si. `SessionService` consulta `ConnectionService` antes de
associar um cliente, mantendo a colaboração na camada de Services. O gateway nunca acessa Maps
ou stores diretamente.

## Protocolo

Todo evento de aplicação utiliza um valor do enum `EventType`. Toda mensagem utiliza o envelope:

```ts
interface SocketMessage<T> {
  id: string;
  event: EventType;
  timestamp: string;
  sessionId?: string;
  payload: T;
}
```

Eventos conhecidos:

- `COMMUNICATION_PING` e `COMMUNICATION_PONG`;
- `SESSION_CREATE` e `SESSION_CREATED`;
- `SESSION_JOIN` e `SESSION_LEAVE`;
- `SESSION_CLOSE` e `SESSION_CLOSED`.

## Fluxo

1. Cada conexão Socket.IO é registrada pelo `ConnectionService`.
2. Um cliente envia `SESSION_CREATE` dentro de `SocketMessage<T>`.
3. `SessionService` cria uma sessão `WAITING` com UUID no `SessionStore`.
4. O servidor responde `SESSION_CREATED` com a sessão criada.
5. Dois clientes enviam `SESSION_JOIN` com o mesmo `sessionId`.
6. `SessionService` valida as conexões e `SessionManager` associa os IDs dos clientes.
7. Com dois clientes, a sessão passa para `ACTIVE`.
8. `SESSION_CLOSE` marca a sessão como `FINISHED`, notifica os clientes e remove a sessão do Map.
9. Na desconexão, o cliente é removido do registro e de qualquer sessão ainda aberta.

## Armazenamento

`ConnectionManager` e `SessionStore` usam `Map`. Nenhum estado é persistido após o encerramento
do processo. O `ConnectionManager` também mantém `lastHeartbeatAt`, preparando a futura política
de heartbeat sem implementá-la nesta sprint.

## Como testar

Execute todas as verificações e testes:

```bash
npm install
npm run check
npm run build
```

O teste de sessão usa dois clientes `socket.io-client` reais e verifica conexão, UUID, associação,
estado `ACTIVE`, encerramento `FINISHED` e remoção do armazenamento.

Para testar somente o workspace WebSocket:

```bash
npm run test --workspace=@professor-connect/websocket
```

Para validar manualmente o ping/pong, execute em terminais separados:

```bash
npm run dev
npm run communication:client
```
