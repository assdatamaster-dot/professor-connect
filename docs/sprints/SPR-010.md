# SPR-010 — WebRTC Signaling Server

## Objetivo

Implementar o transporte de mensagens de negociação entre dois clientes conectados a uma Session
e Call ativas. Esta sprint não cria conexões WebRTC, não interpreta SDP, não captura mídia e não
implementa vídeo, áudio, compartilhamento de tela ou controle remoto.

## Arquitetura

```text
packages/protocol/src/
├── protocol.ts             # EventType
└── signaling.ts            # payloads e códigos de erro
              │
services/backend/services            # Session, Call, Connection e Presence existentes
              │
services/backend/websocket/src/modules/signaling/
├── signaling.events.ts     # aliases tipados para EventType
├── signaling.types.ts      # portas, rotas e tipos Socket.IO
├── signaling.manager.ts    # valida contexto e resolve o destinatário
├── signaling.service.ts    # cria SocketMessage<T> e encaminha
└── signaling.gateway.ts    # recebe eventos e valida o envelope/payload
```

O módulo continua na fronteira WebSocket e depende somente das APIs públicas dos serviços
existentes. Não foi criado banco, estado de sinalização ou acoplamento dos serviços com Socket.IO.

## Fluxo

```text
Cliente A                     Servidor                      Cliente B
   │ signal.offer                │                              │
   ├────────────────────────────>│ valida Session/Call/clientes │
   │                             ├──── signal.offer ───────────>│
   │                             │<─── signal.answer ───────────┤
   │<──── signal.answer ─────────┤                              │
   │                             │                              │
   ├── signal.ice-candidate ────>├── signal.ice-candidate ────>│
   │<─ signal.ice-candidate ─────┤<─ signal.ice-candidate ──────┤
```

Para cada mensagem o gateway:

1. valida `id`, `timestamp`, `event` e `sessionId` do envelope;
2. valida os campos obrigatórios do payload;
3. confirma que a Session existe, está `ACTIVE` e possui exatamente dois clientes;
4. confirma que ambos estão conectados e que o remetente pertence à Session;
5. confirma que a Call existe, não está em estado terminal e corresponde aos participantes;
6. cria um novo `SocketMessage<T>` e o envia somente ao outro cliente.

Se qualquer validação falhar, o remetente recebe `signal.error`; nada é encaminhado ao par. O erro
também é registrado pelo logger.

## Eventos e payloads

Todos os nomes pertencem ao enum `EventType`. Não há strings de eventos espalhadas pelo módulo.

| Enum                   | Valor                  | Payload                     |
| ---------------------- | ---------------------- | --------------------------- |
| `SIGNAL_OFFER`         | `signal.offer`         | `SignalOfferPayload`        |
| `SIGNAL_ANSWER`        | `signal.answer`        | `SignalAnswerPayload`       |
| `SIGNAL_ICE_CANDIDATE` | `signal.ice-candidate` | `SignalIceCandidatePayload` |
| `SIGNAL_ERROR`         | `signal.error`         | `SignalErrorPayload`        |

Offer:

```json
{
  "callId": "call-uuid",
  "sdp": "offer-sdp"
}
```

Answer usa a mesma estrutura, com o SDP de resposta. ICE Candidate:

```json
{
  "callId": "call-uuid",
  "candidate": "candidate-data",
  "sdpMid": "0",
  "sdpMLineIndex": 0,
  "usernameFragment": "optional-fragment"
}
```

Erro:

```json
{
  "code": "CALL_NOT_ACTIVE",
  "message": "Call não está ativa: call-uuid",
  "relatedEvent": "signal.offer"
}
```

Cada payload é transportado dentro de:

```json
{
  "id": "message-uuid",
  "event": "signal.offer",
  "timestamp": "2026-07-20T12:00:00.000Z",
  "sessionId": "session-uuid",
  "payload": {}
}
```

## Logs

O servidor registra `Offer recebida`, `Offer enviada`, `Answer recebida`, `Answer enviada`, `ICE
recebido`, `ICE enviado` e `Erro de sinalização`, sempre com contexto técnico e sem armazenar o
conteúdo SDP ou ICE.

## Como testar

O teste Socket.IO cria dois clientes reais e percorre Presence, Session, Request e Call antes de
simular Offer, Answer e ICE nos dois sentidos. Há também cobertura direta das validações do
manager e do retorno `signal.error`.

```bash
npm run test --workspace=@professor-connect/websocket
npm run check
npm run build
```

## Limites da sprint

Não foram implementados WebRTC, vídeo, áudio, `MediaStream`, screen sharing, remote control,
banco de dados, JWT ou login. SDP e ICE são dados opacos para o servidor e existem somente durante
o encaminhamento da mensagem.
